## 1. DynamoDB / NoSQL

### 1-① 파티션 키 (PK) — ✅ 완료
- 개념 노트: [[DynamoDB 파티션 키]]

**내가 인출한 것**
1. 파티션 키 = 데이터가 어느 파티션에 저장될지 아는 분리키 ✅
2. 카디널리티 높으면 중복 많아 조회 성능 나쁨 ❌
3. 정렬 키 복합으로 시계열 범위 조회 (오늘 데이터만) ✅

**교정 포인트 (틀렸던 것 = 복습 1순위)**
- ❌→✅ **카디널리티 방향을 거꾸로 이해했다.** 높음 = 고유값 많음 = 중복 적음 = 분산 좋음. 파티션 키는 카디널리티가 **높아야** 좋다. (`status` 같은 저카디널리티가 위험)
- 🔍 **내가 스스로 반박해서 더 정밀해진 부분**: "카디널리티 낮으면 무조건 핫 파티션이냐?" → 아니다. 핫 파티션의 직접 원인은 **액세스 skew**. 카디널리티는 필요조건이지 충분조건이 아님. 고카디널리티여도 특정 값에 트래픽 쏠리면 핫 파티션 생김.
- 보강: "분리키"보다 **해시 분산키**. 핫 파티션은 데이터 용량보다 **트래픽 집중**이 1차 원인.

### 1-② 핫 파티션 문제 ⭐🪝 — ✅ 완료
- 개념 노트: [[핫 파티션]]

**내가 인출한 것**
1. write sharding = suffix 추가, 동작/효과는 어렴풋 (정직하게 ? 표시) 🟡
2. 쪼개면 조회 조건이 길어져 사용(읽기) 쪽에 문제 → ✅ 트레이드오프 방향 정확히 짚음

**채운 것 (몰랐던 것 = 복습 1순위)**
- suffix 만드는 법 2가지: **무작위**(쓰기 균등, 읽기 전체 fan-out) vs **계산형**(조회속성 해시 mod N, 단일 읽기는 fan-out 회피). 이 분기를 몰랐음.
- fan-out 메커니즘: Query는 PK 등호 필수 → N개 샤드에 각각 Query → 병렬 실행 → 결과 merge + 재정렬.
- 샤드 수 N = 분산 정도 ↔ 읽기 fan-out 비용. 트레이드오프.
- 1차 방어는 **adaptive capacity**(자동), write sharding은 극단적 집중일 때 수동 대책.

**한 줄 핵심**: write sharding = "쓰기 분산"을 얻고 "읽기 단순함"을 내주는 패턴.

**되물음으로 보강된 것**: "write sharding이 파티션을 나누는 거냐?" → ❌. 개발자는 파티션을 직접 못 나눔. **파티션 키 "값"을 다양화**해 DynamoDB가 분산하도록 유도하는 것(=카디널리티 인위적으로 높이기). 핫 파티션 노트에 ⚠️ 박스로 보강함.

### 1-③ 정렬 키(SK) — ✅ 완료
- 개념 노트: [[정렬 키]]
- (인출 없이 설명 요청) — SK의 4가지 역할 + 계층 데이터 학습.

**배운 것**
- SK 본질 = 같은 PK 안에서 아이템을 **정렬·조직하는 축** (단순 범위쿼리용이 아님).
- 4역할: ① 정렬(ScanIndexForward=false로 최신순) ② 범위조회 ③ 한 파티션에 여러 엔티티 조직화(단일테이블, 아이템 컬렉션) ④ 계층 데이터.
- **계층 데이터**: 구분자 `#`로 계층 이어 저장 → 문자열 정렬로 prefix 인접 → begins_with prefix 길이로 조회 깊이 조절. prefix는 **왼쪽부터 연속**이어야(중간 건너뛰기 불가).
- KeyCondition: PK 등호 필수, SK 선택. **FilterExpression은 읽은 뒤 버려서 RCU 절약 안 됨** (중요 디테일).
