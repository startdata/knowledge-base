# 전량 수집 vs 증분 수집 — 설계 결정 (토스 사전과제 배치)

## 문제 · 배경
토스 사전과제: 10분 주기로 4001/4002/4003 API + CSV에서 Transaction 수집 → 4596으로 StoreTransaction 병합 → MergeTransaction을 DB 저장. **가장 큰 설계 고민 = "증분 수집할 것인가, 매번 전량 재수집할 것인가".**

제약:
- 소스 API는 `?page=N` / `{page}` 등 **offset 페이징만** 제공. 시점 필터(`updated_at>X`) 없음.
- 모의 서버는 런타임 데이터 고정(setInterval/setTimeout 0건 확인). 단 README는 "csv 주기 업로드 가정 / 소스 미래 확장"을 명시 → **데이터가 늘어난다는 전제로 설계**해야 함.

## 접근 · 시행착오 (의사결정 서사)
1. 초기 프레이밍: **"API가 증분 불가능"** 이라고 단정.
2. 반박: "page는 *요청자가* 정하잖아 → 51부터 요청하면 증분 아닌가?" + "1~50 고정 요청인 줄 알았다."
3. 정정 ①(코드): 페이징은 1~50 고정이 아니라 `for(page=1;;page++)` + `hasNextPage`까지 **동적 루프**. 페이지가 늘어도 자동 전량 수집.
4. 정정 ②(개념): **page는 offset(위치)이지 증분 키가 아니다.** "51부터"가 안전하려면 1~50 위치가 안정적이어야 하는데, 정렬·삽입 방식 스펙이 없어 보장 불가 → 누락 위험. ([[offset 페이지네이션]] 불안정성)
5. 최종 정련(가장 정확한 프레이밍): **"불가능"이 아니라 "안전하게 할 스펙 보장이 없으니, 불확실성에 대한 안전한 기본값으로 전량 조회"**.

## 결정
**전량 추출(full extract) + transactionId 멱등 적재(idempotent load).**

### 3중 근거
1. 증분 수단(시점 필터) **부재** — 명세·샘플 기준 사실 (grep의 since/from/sort는 번들 라이브러리 노이즈).
2. page = offset → 위치 불안정 → 유사증분 위험.
3. **안전한 기본값**: 검증 안 된 가정(page 안정성) 위 최적화는 silent data loss. 전량 비용은 분 단위라 감당 가능.

### 비용 상쇄
- 거래 조회는 전량(불가피).
- 비싼 4596 병합([[N+1 문제]])·DB 쓰기는 `findSavedTransactionIds`로 **신규분만** → 2회차부터 0 수렴. = "적게 가져오기" 대신 "적게 처리하기"로 증분 실익 회수.

## "중복 저장 방지" 요구와의 관계 (인과 정정)
- ❌ "중복 방지 요구 → 그러니 전량" (비약)
- ✅ "소스가 증분 미지원 → 전량 → 전량이라 중복 발생 → 멱등 저장이 필수가 됨". **중복 방지는 전량의 *원인*이 아니라 *귀결*.**
- 멱등성은 수집 방식과 독립(제공측 중복·재실행 대비). [[멱등성]]

## 결과 (수치)
- 1회차 99,998건 전량 적재.
- 2회차 신규 저장 0건 → 멱등성 증명.
- 페이지 단위 격리: 2회차 50건 스킵 → 3회차에 **정확히 그 50건만** 신규 저장(= [[최종적 일관성]] 자가치유 증명).

## 검증
- 모의 서버 setInterval/setTimeout 0 → 런타임 데이터 고정 확인.
- grep으로 시점 필터 흔적 조사 → express 등 번들 노이즈, 실 증분 파라미터 없음.
