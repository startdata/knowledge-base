# Redis 분산락 #stub

> `[[최종적 일관성]]` · `[[낙관적 락]]`(아직 노트 없음, 학습 큐) 와 연결.
> 마인드맵 경로: `3. 동시성 & 락 → 분산 직렬화 → Redis 분산락` / `10. Redis → 분산락 ⭐`.
> 팔 노드: SET NX EX / fencing token / Redlock 논쟁.

## 1. 왜 필요한가 (✅ 자기 말로 설명 완료)
서버가 **여러 개**일 때 같은 DB 자원을 동시에 update할 수 있다. 인프로세스 락(자바 `synchronized`, Node.js 인메모리 플래그)은 **그 프로세스 메모리 안에서만** 유효해서 다른 서버의 락을 못 본다. 그래서 모든 서버가 함께 보는 **외부 공유 락**(Redis)이 필요하다. → 공유 자원 접근을 한 번에 하나로 **직렬화**한다.

- 서버가 딱 한 대면 분산락 불필요(인프로세스 락으로 충분).
- 분산락 ≠ 데드락 방지 도구. (자주 혼동하는 지점)

## 2. SET NX EX — 획득과 안전한 해제 (✅ 자기 말로 설명 완료)
**왜 Redis로 락을?** 인메모리라 빠르고, **싱글 스레드로 명령을 원자적으로** 처리한다. `SET NX`·`INCR`·Lua가 경쟁 없이 한 번에 실행돼 락 구현에 적합. (마인드맵 "Redis 싱글 스레드 모델 → 명령 단위 원자성")

```
SET lock_key <UUID> NX EX <초>
```
- **NX** (Not eXists): 키가 없을 때만 SET 성공 → **상호배제**. 이미 누가 잡고 있으면 실패.
- **EX** (expire/TTL): 락 잡은 서버가 **죽어도** 시간이 지나면 자동 삭제 → 영구 점유 방지.
- **value = 매 획득마다 다른 UUID**: 소유권 식별용. "서버 이름"은 부족(같은 서버의 다른 스레드/재획득을 구분 못 함).

**안전한 해제(safe release):** 그냥 `DEL` 하면 안 됨. 내 TTL이 만료된 뒤 남이 새로 잡은 락을 내가 지워버릴 수 있다.
```
풀 때: GET lock_key == 내 UUID ?  → 같으면 DEL, 다르면(=내 건 만료됨) 안 건드림
```
이 `GET→DEL`은 **Lua 스크립트로 원자 실행**해야 함. (GET·DEL 사이에 TTL 만료 + 타인 획득 레이스 방지. 마인드맵 "Lua 스크립트(원자 실행)" 노드)

**TTL 길이 딜레마 (기본):**
- 너무 **짧으면** → 작업이 안 끝났는데 만료 → 다른 클라가 침범(§3 stale write).
- 너무 **길면** → 보유자가 죽었을 때 그만큼 오래 잠김 → 가용성↓.
- 해법 ①: 작업 최대 소요시간보다 약간 길게 잡는다. 해법 ②: **watchdog(lease 자동 연장)** — 클라가 살아있는 동안 백그라운드로 TTL을 주기적으로 갱신해 "작업 중 만료"를 막는다. (예: Redisson 기본 락은 lease 미지정 시 30s TTL + 10s마다 watchdog 갱신.)
- 단, watchdog도 프로세스가 *행(hang)* 되면 갱신이 멈춰 결국 만료되므로 stale write 위험을 완전히 없애진 못함 → 정합성엔 여전히 fencing token.

## 3. TTL의 함정과 fencing token  #stub
> ⚠️ 대화로 이해는 했으나 **자기 말로 재인출 미완료** — 나중에 `/학습`으로 다시 인출할 것.

**TTL은 양날의 검.** 락 잡은 서버 A가 *죽은 게 아니라* 잠깐 멈추면(GC stop-the-world, 네트워크 지연) 그 사이 TTL이 만료된다. A는 **그 사실을 통보받지 못한다**:
- A는 처음 `SET NX` 성공 순간의 **메모리 속 믿음**("난 락 보유 중")으로 작업을 끝까지 진행한다. 작업(DB write) 중엔 Redis를 다시 쳐다보지 않는다.
- Redis는 TTL 만료를 A에게 push로 알리지 않는다.
- 그 사이 B가 락을 새로 잡음 → **A·B 둘 다 락 보유** → 동시 쓰기 충돌(stale write).
- (설령 A가 "나 락 있어?"라고 물어도, "있어" 응답 직후 update 실행 찰나에 만료되면 소용없음 = check-then-act 레이스, TOCTOU)

**fencing token = 자원 쪽에서 막는다.**
- 락 잡을 때마다 **1씩 증가하는 번호표** 발급(Redis `INCR`).
- 클라이언트가 자원(DB)에 쓸 때 **번호표를 같이** 보냄. (Redis는 발급만; 중간 프록시 아님. 앱이 Redis·DB에 각각 직접 말 검)
- 자원은 **지금까지 본 최댓값**을 기억하고: `들어온 번호 > 최댓값`이면 수락+갱신, 아니면 **거부**.
- → 좀비 A(낡은 번호 33)가 늦게 와도, DB가 이미 34를 봤으면 거부. **A의 생사를 확인할 필요 없이 번호 순서만** 본다.

**자원 쪽 저장 = 낙관적 락의 version 컬럼과 동일한 모양.**
```sql
UPDATE account SET balance=?, last_token=33 WHERE id=? AND last_token < 33;
-- 영향 0행이면 거부
```
차이: 고전 낙관적 락은 version을 **DB/ORM이 매 쓰기마다 자동 증가**(읽기-쓰기 충돌 감지). fencing token은 번호 출처가 **외부 락 서비스(Redis INCR)**, "가장 최근 락 보유자만 쓴다"는 **락 획득 순서**를 강제. 자원 쪽 메커니즘(단조 증가 컬럼 + 조건부 쓰기)은 같음.

## 4. Redlock 논쟁  #stub
> ⚠️ 대화로 이해는 했으나 **자기 말로 재인출 미완료** — 나중에 `/학습`으로 다시 인출할 것.

**단일 Redis의 위험(safety):** 마스터-레플리카는 **비동기 복제**. 마스터가 락을 주고 레플리카로 복제하기 *전에* 크래시 → failover로 승격된 레플리카엔 그 락이 없음 → B가 또 잡음 → A·B 둘 다 보유. (복제 지연이 뿌리, `[[최종적 일관성]]`)

**Redlock 알고리즘:** 서로 **독립된**(복제 안 하는) Redis **N대**에 동시에 락 시도 → **과반(N/2+1)** 에서 성공해야 락 획득.
- 왜 과반? 두 클라이언트가 동시에 과반을 차지할 수 없음 — N대 중 두 과반 집합은 **반드시 최소 1대에서 겹치고**, 그 1대는 한 명에게만 줌 → 상호배제. 소수 노드가 죽어도 과반이 살면 유지 → SPOF 해소.

**논쟁:**
- **Kleppmann 비판:** Redlock도 GC 정지·네트워크 지연·시계 점프(clock drift) 앞에선 좀비의 지각 쓰기를 못 막음. 진짜 안전하려면 **fencing token 필수**인데 Redlock은 단조 번호표를 깔끔히 못 줌 → 정합성 중요하면 ZooKeeper 등 합의 시스템 써라.
- **antirez(Redis 창시자) 반박:** 가정을 명확히 하면 충분히 쓸 만하다.
- **결론(takeaway):** 락 용도를 나눠라.
  - **효율성용**(중복 작업 방지, 어쩌다 두 번 실행돼도 손해만): 단일 Redis 락으로 충분.
  - **정합성용**(두 번 실행되면 데이터 깨짐): Redlock만으론 부족, **fencing token 필요**.

## 5. 순서(fairness) 보장 안 함
- `SET NX` 락은 **대기 큐가 없다.** 기다리는 클라들이 주기적으로 **재시도(폴링)** 하다가, 락 풀리는 순간 **타이밍 먼저 맞은 놈**이 잡음 → FIFO 아님 = **공정성 없음**.
- "데드락 막는다"는 *TTL 덕에 영구 무한대기가 없다*는 좁은 의미. 여러 락을 다른 순서로 잡으면 순환대기 가능(단 TTL로 자연 해소).
- 순서가 필요하면 락이 아니라 **단일 소비자 직렬 처리**(같은 키→한 파티션→한 컨슈머)가 정석. (마인드맵 "Kafka 파티션 순차 처리 ⭐")
- 방증: Redisson은 기본 락이 불공정해 별도 **Fair Lock**(내부 큐)을 제공.

## 학습 중 내가 던진 질문 (막혔던 지점 Q&A)
이해가 갈렸던 실제 질문들. 다시 막히면 여기부터 본다.

- **Q. TTL로 락이 삭제되면 A는 "나 락 없다"는 걸 아는 거 아냐?**
  A. 모른다. A는 작업 중 Redis를 다시 안 본다(처음 성공 순간의 메모리 속 믿음으로 진행). Redis도 만료를 push로 안 알림. → 이게 §3 함정의 핵심.

- **Q. fencing 흐름에서 Redis가 번호표 확인하고 DB로 요청을 넘겨주는 거야?**
  A. 아니다. Redis는 **발급만**(INCR), 검사는 **DB가** 한다. 앱이 Redis·DB에 *각각 직접* 말 검(Redis는 프록시 아님). DB는 A의 생사 확인 없이 **번호 순서만** 본다.

- **Q. UUID는 서버를 식별 못 하는데, 그냥 "새 거구나" 하는 거야?**
  A. 맞다. 누가 잡았는지는 관심 없음. 풀 때 "Redis의 락이 **내가 만든 그 인스턴스인가**"만 판별 → 값 다르면 내 건 만료된 것 → 안 건드림.

- **Q. 단조증가 번호를 어디에 기록해? 별도 테이블?**
  A. 발급(증가)은 **Redis INCR**, 최댓값 기억(검증)은 **보호 자원(DB) 쪽 컬럼**. 별도 테이블 아니고 보통 지키려는 레코드 옆. → 낙관적 락 version 컬럼과 같은 모양.

- **Q. 그건 낙관적 락처럼 version 컬럼을 따로 두는 거야?**
  A. 그렇다. 자원 메커니즘(단조증가 컬럼 + 조건부 쓰기)은 동일. 다른 건 번호 **출처**(외부 락 서비스 vs DB 자동 증가)뿐.

- **Q. 분산락이 데드락은 막지만 순서는 보장 안 한다는데 왜?**
  A. "데드락 막음"은 TTL 덕에 영구 무한대기가 없다는 좁은 의미. 순서 미보장은 **대기 큐 없이 재시도 경쟁**이라 FIFO/공정성이 없어서. 순서 필요하면 단일 소비자 직렬 처리. (§5)

## 복습 질문 #flashcards
- 서버 한 대면 분산락이 필요 없는 이유는? :: 인프로세스 락이 그 프로세스 메모리 안에서만 유효하기 때문. 여러 서버일 때만 외부 공유 락 필요.
- `SET NX EX`에서 EX(TTL)가 필요한 이유는? :: 락 잡은 서버가 죽으면 영구 점유되므로 시간 지나면 자동 해제하려고.
- 락 value에 UUID를 넣는 이유는? :: 풀 때 "Redis의 락이 내가 만든 그 인스턴스인가" 식별 → 남의 락을 지우지 않기 위해.
- `GET→DEL` 해제를 Lua로 묶는 이유는? :: 두 명령 사이에 TTL 만료+타인 획득 레이스가 끼어들 수 있어서(원자 실행 필요).
- TTL의 함정은? :: 보유자가 안 죽고 멈췄을 때(GC 등) TTL 만료를 통보받지 못해 A·B 둘 다 락 보유 → stale write.
- fencing token이 stale write를 막는 원리는? :: 락마다 단조증가 번호 발급, 자원이 최댓값보다 작은 번호의 쓰기를 거부.
- Redlock에서 과반이 필요한 이유는? :: 두 과반 집합은 최소 1대에서 겹치고 그 노드는 한 명에게만 락을 줘서 상호배제가 성립하므로.
- 분산락이 순서를 보장 안 하는 이유는? :: 대기 큐가 없고 재시도(폴링) 타이밍 경쟁이라 FIFO/공정성이 없음.
