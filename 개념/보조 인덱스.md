# 보조 인덱스 (Secondary Index) — GSI / LSI

DynamoDB에서 **파티션 키가 아닌 속성으로 조회**할 수 있게 해주는 추가 조회 경로.

## 왜 필요한가
- DynamoDB Query는 **파티션 키 등호(`=`)가 필수** → [[DynamoDB 파티션 키]]가 아닌 속성으로는 조회 불가. 유일한 대안이 Scan(전체 훑기, 비쌈).
- 보조 인덱스 = **다른 속성을 키로 하는 또 다른 조회 경로**를 만들어 새 액세스 패턴을 추가.
  - 예: 테이블 PK가 `user_id`인데 `email`로 찾고 싶다 → `email`을 PK로 하는 GSI.
- ⚠️ **못 푸는 것**: `group by`, `sum`, `count` 같은 SQL식 **집계는 보조 인덱스로도 안 된다.** DynamoDB가 원래 미지원 → 애플리케이션/집계 테이블/스트림+Lambda 몫. 보조 인덱스는 "다른 키로 Query"이지 집계가 아니다.

## GSI vs LSI

| | **GSI** (Global) | **LSI** (Local) |
| --- | --- | --- |
| PK | **다르게** 지정 가능 | 테이블과 **동일** (SK만 다름) |
| 생성 시점 | **언제든** 추가/삭제 | **테이블 생성 시에만**, 이후 불가 |
| 읽기 일관성 | **eventually only** | **strong 가능** |
| 저장/용량 | 별도 공간, **별도 RCU/WCU** | 테이블과 **같은 파티션, RCU 공유** |
| 제약 | — | 파티션당 **10GB** (아이템 컬렉션 크기 제한) |

### 일관성 차이의 이유 (핵심 인과)
- **LSI는 테이블과 같은 파티션**에 저장 → 강한 일관성 읽기 가능. 같은 PK를 쓰고 SK만 다르니 같은 파티션에 얹힘.
- 같은 파티션 구조에 얹혀야 하므로 **파티션 구조가 정해지는 테이블 생성 시점에 고정** → 나중에 추가 불가, PK 변경 불가.
- **GSI는 별도 공간에 비동기 복제** → 복제 지연 존재 → eventually consistent only. 대신 PK 자유·언제든 추가 가능.

### 실무 결론
- 대부분 **GSI 사용** (유연, 나중 추가, PK 자유).
- LSI는 제약(생성 시 고정·10GB·PK 동일)이 많아 "같은 PK + 다른 정렬축 + 강한 일관성"이 꼭 필요할 때만.

## GSI 심화 3종
- **projection** (인덱스에 복사할 속성): `KEYS_ONLY`(키만) / `INCLUDE`(지정) / `ALL`(전부). 인덱스에 없는 속성 조회 시 테이블 재조회 필요(비용↑). 저장공간 ↔ 조회편의 트레이드오프.
- **sparse index**: 인덱스 키 속성이 **있는 아이템만** 인덱스에 포함. 예: `unprocessed`를 미처리 주문에만 달면 그 GSI엔 미처리 주문만 모임 → "미처리만 조회"가 작고 빠른 인덱스로 해결.
- **index overloading**: 하나의 GSI를 여러 엔티티가 공유. 키를 범용 이름(`GSI1PK`, `GSI1SK`)으로 두고 엔티티마다 다른 의미 부여 → 단일 테이블 설계에서 인덱스 수 절약.

#flashcards
- Q: 보조 인덱스가 푸는 문제와 못 푸는 문제는? → A: 푸는 것 = PK 아닌 속성으로 조회(다른 키 Query 경로 추가). 못 푸는 것 = group by/sum 같은 집계.
- Q: GSI와 LSI 중 강한 일관성 읽기가 가능한 것은? 왜? → A: LSI. 테이블과 같은 파티션에 저장되기 때문. GSI는 별도 공간 비동기 복제라 eventually only.
- Q: LSI를 테이블 생성 시점에만 만들 수 있는 이유는? → A: 테이블과 같은 PK·같은 파티션에 얹혀야 해서 파티션 구조가 정해지는 생성 시점에 고정.
- Q: GSI projection 3종은? → A: KEYS_ONLY / INCLUDE / ALL. 인덱스에 없는 속성 조회 시 테이블 재조회 비용 발생.
- Q: sparse index란? → A: 인덱스 키 속성이 있는 아이템만 인덱스에 포함되는 성질. 부분집합만 모은 작은 인덱스로 특정 조회를 최적화.
- Q: 실무에서 GSI를 더 많이 쓰는 이유는? → A: PK 자유·언제든 추가 가능 등 유연성. LSI는 생성 시 고정·10GB·PK 동일 제약이 많음.
