# 테스트 디렉토리 구조 (co-located vs 별도 tests/)

## 두 방식
- **co-located**: 소스 옆에 테스트를 나란히. `foo.ts` + `foo.spec.ts`.
- **별도 `tests/`**: `src`와 같은 계층의 `tests/` 폴더에 `src` 구조를 미러링. `tests/batch/collect/...`.

## 비교
| 기준 | co-located | 별도 `tests/` |
|---|---|---|
| 테스트↔소스 근접성 | ★ 바로 옆, 즉시 발견 | 미러 경로 따라가야 |
| 리팩터링 시 동반 이동 | ★ 파일 옮기면 테스트도 함께 | 두 트리를 따로 동기화 |
| 소스 트리 순수성 | 테스트가 섞임 | ★ src가 프로덕션 코드만 |
| 빌드 산출물 제외 | `.spec` 제외 설정 필요 | ★ `tests/` 통째 제외 쉬움 |
| 단위 vs 통합/E2E | 단위에 최적 | ★ 통합·E2E 분리 자연스러움 |
| 생태계 관례 | JS/TS(Jest·Vitest), React | Java(Maven `src/test`)·Python·.NET |

## 트레이드오프 핵심
- **co-located**: 근접성(테스트가 명세 역할) + 리팩터링 동반 이동.
- **별도 `tests/`**: 소스 트리 순수성 + 빌드 제외 용이 + 통합/E2E 계층 분리.

## 맥락별 선택
- JS/TS(Jest·Vitest)·단위 중심·React → co-located 관례.
- Java·Python·.NET → 별도 디렉토리 관례.
- **NestJS 기본값 = 하이브리드**: 단위는 소스 옆 `.spec.ts`, E2E는 별도 `test/` 폴더.

## 권장
**하이브리드** — 단위는 소스 옆(co-located), 통합·E2E는 별도 `test/`. 단위까지 전부 `tests/`로 빼면 근접성·동반 이동 이점을 잃는다. 절대 우열은 없고 테스트 성격·스택·팀 컨벤션에 의존.
