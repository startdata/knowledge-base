# DynamoDB 파티션 키

DynamoDB에서 아이템이 **어느 물리 파티션에 저장될지** 결정하는 키. "분리키"보다 **해시 분산키**가 정확한 표현이다.

## 메커니즘 — 해시 분산
- 파티션 = 데이터가 실제로 저장되는 물리 단위 (SSD, 한 파티션 ≈ 10GB / 3000 RCU / 1000 WCU 상한).
- 쓰기·읽기 시 DynamoDB가 **파티션 키 값을 내부 해시 함수에 통과**시켜, 그 해시값으로 저장/조회할 파티션을 정한다.
- 따라서 파티션 키 값이 **다양할수록** 해시 결과가 골고루 흩어져 여러 파티션에 균등 분산된다.

## 카디널리티 — 높을수록 좋다
카디널리티 = 그 속성이 가질 수 있는 **고유값의 다양성(개수)**.

| 구분 | 의미 | 예시 | 파티션 키로 |
| --- | --- | --- | --- |
| 높은 카디널리티 | 고유값 많음 = 중복 적음 | `user_id`, `order_id` | ✅ 좋음 (균등 분산) |
| 낮은 카디널리티 | 고유값 적음 = 몰림 | `status`(active/inactive), `country`(KR) | ❌ 나쁨 (한 파티션 집중) |

- 흔한 오개념: "카디널리티 높으면 중복 많아 느리다" → **거꾸로다.** 높음 = 중복 적음 = 분산 좋음.
- 저카디널리티 속성을 꼭 키로 써야 하면: 고카디널리티 속성과 **복합**(`ACTIVE#customer123`)하거나 write sharding으로 쪼갠다.

### 카디널리티 ≠ 핫 파티션의 직접 원인 (정밀화)
핫 파티션의 직접 원인은 카디널리티가 아니라 **액세스 분포의 불균형(skew)**이다. 카디널리티는 균등 분산의 **필요조건이지 충분조건이 아니다.**

| 카디널리티 | 액세스 분포 | 핫 파티션? |
| --- | --- | --- |
| 낮음 | 균등 | 안 생길 수 있음 (단 값이 N개면 최대 N그룹 — 처리량 상한 막힘) |
| 낮음 | 편중 | 생김 (위험 높음) |
| 높음 | 균등 | 안 생김 (이상적) |
| 높음 | 편중 | **생김** (예: `user_id`가 키여도 유명인 1명에 트래픽 폭주) |

- 즉 **고카디널리티여도 특정 값에 트래픽이 쏠리면 핫 파티션**이 생긴다. 카디널리티 높음 = 안전 보장 아님.
- 저카디널리티가 위험한 이유는 "반드시 핫"이 아니라 **분산 상한을 구조적으로 막아** skew에 취약하기 때문.
- 핫 파티션 = 특정 파티션에 **트래픽**이 몰려 처리량 한도(~3000 RCU / 1000 WCU)를 초과해 스로틀링. (데이터 용량보다 **액세스 집중**이 1차 원인) → [[핫 파티션]]
- DynamoDB의 **adaptive capacity**가 핫 파티션을 자동 감지해 처리량을 재분배·격리하지만 한도가 있어 극단적 skew는 못 막는다.

## 복합 키 (PK + SK)
- 파티션 키만으로는 같은 PK의 **아이템 컬렉션 전체**만 조회 가능 (PK는 항상 등호 `=`로 정확히 지정).
- **정렬 키(SK)**를 더하면 같은 PK 안에서 범위를 좁힐 수 있다 → [[정렬 키]].
  - 예: `PK=user_id`, `SK=timestamp` → `between '2026-06-22T00' and '2026-06-22T23'` 로 오늘 데이터만.
  - SK 연산자: `begins_with`(계층), `between`(범위), `>`,`<` 등. 모두 `KeyConditionExpression`에 들어간다.
- 중요: **SK 범위 쿼리는 같은 PK 안에서만** 동작한다. 파티션을 가로지르는 범위 조회는 불가.

#flashcards
- Q: 파티션 키는 어떻게 저장 위치를 정하나? → A: 값을 내부 해시 함수에 통과시켜 나온 해시값으로 파티션 결정.
- Q: 파티션 키의 카디널리티는 높아야 좋은가 낮아야 좋은가? 왜? → A: 높아야 좋다. 고유값이 많아야 해시가 골고루 흩어져 균등 분산된다. (낮으면 분산 상한이 막혀 핫 파티션에 취약해진다)
- Q: 카디널리티가 낮으면 무조건 핫 파티션이 생기나? → A: 아니다. 직접 원인은 액세스 분포의 skew. 저카디널리티여도 액세스가 균등하면 안 생길 수 있고, 고카디널리티여도 특정 값에 트래픽이 쏠리면 생긴다. 카디널리티는 필요조건이지 충분조건이 아니다.
- Q: 핫 파티션의 1차 원인은 데이터 용량인가 트래픽인가? → A: 트래픽(액세스) 집중. 특정 파티션이 처리량 한도(~3000 RCU/1000 WCU)를 초과해 스로틀링되는 것.
- Q: SK 범위 쿼리의 전제 조건은? → A: 같은 PK 안에서만 동작. PK는 등호로 정확히 지정해야 한다.
