# LLM 응답 검증 전략

LLM/에이전트 출력을 검증할 때 **무엇을 어떤 방법으로 검증하느냐**를 고르는 원칙.

## 핵심 원칙: 정답 유무로 방법을 가른다
- **정답(기대값)이 있는 것 → 결정적 코드 비교** (LLM judge 아님)
  - 예: 추출한 파라미터, 선택한 도구, 호출한 API 경로, 상태코드.
  - LLM judge로 하면 비결정적이라 부정확. 코드로 기대값과 비교하면 정확·무료·재현 가능.
- **주관적 품질 → LLM judge** ([[LLM-as-a-Judge]])
  - 예: 정확성/도움됨/완전성/간결성/근거성.

## 시점: 런타임 vs 오프라인
- **런타임(프로덕션 trace)**: 정답 라벨이 없음 → 주관 품질은 LLM judge, 기술적 성공(도구성공/거부/오류)은 코드 score.
- **오프라인(dataset)**: 사람이 기대값을 미리 정함 → 정답 대비 결정적 검증 가능. 회귀테스트·버전 비교에 적합. dataset을 YAML 등 외부 파일로 두면 비개발자도 케이스 추가 가능.

## 에이전트 파이프라인 단계별 검증 (도구 호출 에이전트 예)
1. **파라미터 추출** → 코드(기대 args 비교)
2. **도구/노드 선택** → 코드(기대 tool 비교)
3. **API 호출 정확·성공** → 코드(기대 path + status). ← 이걸 검증하려면 **API 호출을 trace span으로 노출**해둬야 함(안 그러면 도구 함수 내부에 묻혀 안 보임).
4. **결과 요약 품질** → LLM judge.

→ 각 단계를 **trace observation으로 분리**해두면 "어디서 깨졌는지"(파라미터? 도구? API? 요약?)가 단계별로 보인다.
