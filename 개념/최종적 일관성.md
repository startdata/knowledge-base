# 최종적 일관성 (Eventual Consistency)

## 한 줄
즉시 일관성은 포기하되, **시간이 지나면 반드시 일관된 상태에 도달**한다는 보장.

## 배치에서의 모습 — 자가 치유(self-healing)
부분 실패(페이지/그룹 단위)를 스킵해도, **다음 주기 재수집**이 빠진 부분을 메운다.
- 스킵된 건 저장 안 됨 → 다음 주기에 "신규"로 분류되어 다시 수집·저장.
- 별도 보정 작업(reconciliation) 없이 **반복 + 멱등**으로 치유.

## 3가지 전제
1. [[멱등성]] (다시 저장해도 안전)
2. 주기성 (다음 실행이 보장됨)
3. 소스가 그 데이터를 계속 제공 (사라지지 않음)

## stateless 설계 — 상태 대신 결과
"무엇을 했나"를 **체크포인트(상태)**가 아니라 **결과(이미 저장된 id 집합)**로 판단.
- 비유: 책갈피(체크포인트) 대신, 매번 처음부터 훑되 외운 건 건너뛰기. **책갈피가 없으니 잃을 일도 없다.**
- → 상태 관리 0, 복구 단순.

## ⚠️ 한계
**영구 실패는 조용히 누락**(silent data loss). 매 주기 계속 실패하는 건은 warn만 남기고 영원히 스킵될 수 있다.
- 보강: 실패 나이/횟수 추적 → 임계 초과 시 알림(alert) 또는 [[DLQ]]로 격리. "미완성" 레코드를 pending 상태로 가시화.

## 즉시 vs 최종 일관성 trade-off
- 즉시 일관성: 이번 실행에 100% 정합. 복잡·비쌈.
- 최종적 일관성: 단순·복원력↑, 대신 "일시적 불일치 창"(이 과제는 최대 1주기) 허용.

- Q: 배치 자가치유의 3전제? → A: 멱등성 + 주기성 + 소스의 지속 제공.
- Q: stateless 설계가 "무엇을 했나"를 판단하는 기준? → A: 체크포인트가 아니라 이미 저장된 결과(id 집합).
- Q: 최종적 일관성의 한계와 보강책? → A: 영구 실패의 silent loss → 실패 추적 + 알림/DLQ.
