# 핫 파티션 (Hot Partition)

DynamoDB에서 특정 파티션에 **읽기/쓰기 트래픽이 몰려**, 그 파티션의 처리량 한도(~3000 RCU / 1000 WCU)를 초과해 **스로틀링(ProvisionedThroughputExceeded)** 되는 상태.

## 원인 — 데이터 용량이 아니라 액세스 skew
- 1차 원인은 **트래픽(액세스) 집중**이지 데이터 용량이 아니다.
- [[DynamoDB 파티션 키]]의 카디널리티가 낮으면 분산 상한이 막혀 취약하지만, **카디널리티는 필요조건일 뿐** — 고카디널리티여도 특정 값에 트래픽이 쏠리면(유명인 user_id 등) 핫 파티션이 생긴다.
- 전형적 예: **날짜 파티션 키** (`2026-06-22` 하나에 그날 모든 쓰기 집중).

## 1차 방어 — Adaptive Capacity
- DynamoDB가 핫 파티션을 자동 감지해 처리량을 재분배·격리(isolation)해준다.
- 즉각·자동이지만 **한도가 있어 극단적 skew는 못 막는다.** 그 다음 수동 대책이 write sharding.

## 수동 대책 — Write Sharding
핫한 파티션 키 값 뒤에 **suffix를 붙여 N개 샤드로 쪼개** 쓰기를 분산한다.
```
2026-06-22  →  2026-06-22.1, 2026-06-22.2, ... , 2026-06-22.N
```
- **샤드 수 N = 분산 정도.** 클수록 분산↑ 이지만 읽기 fan-out 비용↑.

> ⚠️ **흔한 오해: "파티션을 직접 나누는 것"이 아니다.** 물리 파티션의 분할·개수는 DynamoDB가 내부 관리하며 개발자에게 노출되지 않는다. write sharding이 하는 건 **파티션 키 "값"을 다양화**하는 것뿐 — `2026-06-22.1`과 `.2`는 DynamoDB 입장에서 서로 다른 키 값이고, 다른 해시 → 다른 파티션에 갈 *가능성*이 높아진다. 즉 **카디널리티를 인위적으로 높이는 트릭**이다. 샤드 N개가 정확히 물리 파티션 N개에 1:1 매핑되는 건 보장되지 않는다(해시라 확률적).

### suffix 결정 방식 2가지
| 방식 | suffix | 쓰기 분산 | 읽기 |
| --- | --- | --- | --- |
| 무작위(random) | 1~N 랜덤 | 균등(최고) | 위치 모름 → 전체 fan-out 필요 |
| 계산형(calculated) | 조회 속성 해시 `mod N` | 키 분포 의존 | 위치 역산 가능 → 단일 아이템은 fan-out 불필요 |

### 트레이드오프 — 읽기 복잡도 ↑ (fan-out)
DynamoDB Query는 파티션 키를 **등호(`=`)로 정확히** 줘야 한다. 샤딩하면 한 논리 키의 데이터가 N개 샤드에 흩어지므로, **전체 조회 시:**
1. N개 샤드에 각각 Query (**fan-out**) — 보통 병렬 실행
2. N개 결과를 애플리케이션에서 **병합(merge) + 재정렬**

→ 읽기 비용↑, 레이턴시(가장 느린 샤드가 좌우), 코드 복잡도↑.

## 적용 판단
- **잘 맞음**: 쓰기는 한 키에 폭주 + 읽기는 개별 아이템 위주 (계산형 suffix로 fan-out 회피).
- **부적합**: 그 파티션을 통째로 자주 조회하는 패턴 (fan-out 비용이 이득 상쇄).
- 핵심: **"쓰기 분산"을 얻는 대신 "읽기 단순함"을 내주는** 패턴. GSI에도 동일 적용.

#flashcards
- Q: 핫 파티션의 1차 원인은? → A: 데이터 용량이 아니라 트래픽(액세스) 집중으로 파티션 처리량 한도를 초과한 스로틀링.
- Q: write sharding은 무엇을 얻고 무엇을 내주나? → A: 쓰기 분산을 얻고 읽기 단순함을 내준다(전체 조회 시 fan-out + merge).
- Q: 무작위 suffix와 계산형 suffix의 읽기 차이는? → A: 무작위는 위치를 몰라 전체 fan-out 필요, 계산형은 조회 속성을 해시해 위치를 역산하므로 단일 아이템은 fan-out 불필요.
- Q: 샤드 수 N을 늘리면? → A: 쓰기 분산은 좋아지지만 전체 조회 시 fan-out하는 Query 수가 늘어 읽기 비용·복잡도가 커진다.
- Q: write sharding 전에 DynamoDB가 자동으로 해주는 1차 방어는? → A: adaptive capacity(핫 파티션 자동 감지·처리량 재분배). 단 한도가 있어 극단적 skew는 못 막음.
