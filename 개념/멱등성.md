# 멱등성 (Idempotency)

## 한 줄
같은 연산을 여러 번 수행해도 결과가 한 번 한 것과 같음. 배치는 재실행이 일상이라 **필수**.

## 왜 배치에 필수인가
장애 재시작·중복 트리거·수동 재실행이 일상. 멱등하지 않으면 재실행이 데이터를 오염시킨다.

## 구현: 검사 vs 키 설계
- **검사(read-then-write)**: `find` 후 없으면 `save`.
  - 약점 ① 조회와 저장 사이 경쟁 조건(다른 쓰기 끼어듦) ② 중복 판정 로직이 늘수록 버그 표면 증가.
- **키 설계(권장)**: **자연키**를 저장 키로 → 같은 데이터는 같은 자리에 덮어쓰기.
  - 멱등성이 *로직*이 아니라 *구조*에서 나온다.
  - 예: `transactionId`(UUID)를 키로 upsert → 몇 번 저장해도 결과 동일.

## upsert
insert-or-update를 원자적으로. RDB는 `ON CONFLICT` / `MERGE`, file DB(node-json-db)는 키 경로 덮어쓰기(`push(path, data, false)`).

## 자연키 vs 대리키
- **자연키(비즈니스 키)**: 데이터 자체의 고유 식별자(transactionId). **중복 판정 기준은 항상 자연키.**
- 대리키(surrogate): auto-increment 등 시스템 생성. 중복 판정엔 부적합.

## 수집 방식과 독립
[[증분 수집]]을 해도 멱등은 필요하다 — 제공 측이 중복을 줄 수도, 배치가 재실행될 수도 있으니. **멱등성은 수집 방식과 무관한 기본 안전성.**

## 관련
[[증분 수집]] · [[최종적 일관성]] · [[재시도 전략]] (재시도 안전성 = 요청의 멱등성)

#flashcards
- Q: read-then-write가 멱등 보장에 약한 이유? → A: 조회-저장 사이 경쟁 조건 + 검사 로직의 버그 표면.
- Q: 멱등성을 "구조"로 만드는 방법? → A: 자연키를 저장 키로 한 upsert.
- Q: 증분 수집을 하면 멱등성이 불필요한가? → A: 아니다. 제공측 중복·재실행 때문에 여전히 필수.
