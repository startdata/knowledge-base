# 증분 수집 (Incremental Extraction)

## 한 줄
데이터를 매번 전체 다시 가져오는 대신(full extract), **지난번 이후 변경된 것만** 가져오는 수집 전략.

## 전량(full) vs 증분(incremental)
- **전량 재수집**: 매 실행마다 소스 전체를 다시 읽음. 단순·stateless. 양이 많음.
- **증분 수집**: "마지막 처리 이후"만 읽음. 효율적이지만 "마지막 이후"를 식별할 **안정적 수단**이 필요(stateful).

## 증분의 전제 — 안정적 식별 수단
증분이 성립하려면 "지난번 이후"를 가리킬 **안정적 키**가 있어야 한다:
- 시점 필터: `updated_at > :watermark`
- 단조 증가 id / 안정적 커서(keyset pagination)
- 변경 로그: [[CDC]] (DB 트랜잭션 로그 기반)
→ 구현 기법은 [[워터마크]] 참고.

## ⚠️ page(offset)는 증분 수단이 아니다
`?page=N` 같은 offset 페이징은 **위치 기반**이지 변경분 식별자가 아니다.
- "지난번 50까지 했으니 51부터"가 안전하려면 1~50의 *내용*이 그대로여야 함(위치 안정성).
- 소스가 **최신순 정렬·중간 삽입**을 하면 page 경계가 밀려 → 누락/중복. (= [[offset 페이지네이션]] 불안정성)
- 정렬·추가 방식 스펙이 보장 안 되면 page 기반 유사증분은 위험.

## 전량 + 멱등을 택하는 근거 (실무 판단)
1. 소스가 시점 필터 등 증분 수단을 **미제공**
2. page는 위치 불안정 → 유사증분 위험
3. 규모가 전량 재수집을 **감당 가능**(분 단위)
4. **안전한 기본값**: 검증 안 된 가정(page 안정성) 위 최적화는 silent data loss 위험 → 안전한 단순 설계를 기본으로 깔고, 증분은 스펙이 보장될 때 도입

## 핵심 통찰
증분의 본질은 **"적게 가져오기"**. 전량 + [[멱등성]]은 "많이 가져오되, 비싼 처리(병합·저장)만 신규분으로" 제한 → **'적게 처리하기'로 실익을 회수**한다. 비싼 단계에서만 증분 효과.

> 전량은 "증분이 *불가능*해서"가 아니라 **"증분을 *안전하게* 할 스펙 보장이 없어서"** 택하는 것. ("불가능" 단정은 부정확 — page는 정할 수 있으나 안정적 증분 키가 아닐 뿐.)

#flashcards
- Q: 증분 수집이 성립하기 위한 전제는? → A: "지난번 이후"를 가리킬 안정적 식별 수단(시점/단조 id/커서/CDC).
- Q: page(offset)를 증분 키로 쓰면 안 되는 이유? → A: 위치 기반이라 정렬·삽입 시 경계가 밀려 누락/중복.
- Q: 전량+멱등을 택하는 4가지 근거? → A: 증분 수단 부재 / offset 불안정 / 전량 감당 가능 / 안전한 기본값.
