# 헥사고날 아키텍처 (포트-어댑터)

애플리케이션 핵심(도메인·유스케이스)을 **포트(추상 인터페이스)** 뒤에 두고, 외부 연동(DB·API·UI)을 **어댑터**로 갈아끼우는 구조. 목적: 외부 변경이 핵심에 파급되지 않게 하는 **느슨한 결합**.

## 레이어
- **domain**: 비즈니스 규칙·용어·상태 (외부 무의존, 가장 안쪽)
- **application**: 유스케이스·오케스트레이션 (포트에만 의존)
- **adapters / infrastructure**: 외부 연동 구현 (DB·API·LLM·관측) — 포트를 구현
- **interface / client**: 인바운드 어댑터 (슬랙·웹·CLI 등, "여러 클라이언트 중 하나")
- 의존 방향: `interface → application → domain ← infrastructure`. domain이 중심, 조립은 진입점(composition root)에서 주입.

## AI 에이전트에서: "도구(tool) = 포트"
- 전통 앱은 조회 추상화로 Repository 인터페이스를 포트로 만든다.
- **LLM 에이전트에선 LangChain `tool`이 이미 그 추상화 경계**다. LLM이 도구를 골라 호출하고, 도구 뒤가 REST API든 DB SQL이든 에이전트는 모른다.
- → **도구셋을 갈아끼우면 조회 소스 교체**가 됨(포트-어댑터가 공짜에 가깝게). 도메인 로직의 상당수가 코드가 아니라 **프롬프트+도구 스키마(자연어)** 에 있는 게 일반 앱과 다른 점.

## 트레이드오프: 레이어별(수평) vs 수직 분할
- **레이어별**(domain/app/infra 4계층): 관심사·의존 통제 명확. 근데 한 기능이 여러 폴더에 흩어지고, **소규모(~2천 줄)엔 과설계**일 수 있음.
- **수직 분할**(기능/도메인별 슬라이스, 프롬프트 co-locate): 응집도↑, 격리·"만들며 개선"에 유리. 공통 위치가 애매할 수 있음.
- **실전 절충**: 수직 분할을 기본으로 + **공통만 얇게 domain 추출**. 4계층을 강제하지 않음.
- **"뼈대 vs 살"**: 레이어 경계·의존 방향·플러그인 레지스트리 같은 **뼈대는 처음에 제대로**(나중에 바꾸면 import 전면 수정=비쌈). 정확도·프롬프트·세부 로직 같은 **살은 만들며 반복 개선**.

## 부수 지식
- 파이썬 **src 레이아웃**: 앱은 하나의 루트 패키지 아래 두는 게 관례(`import app.domain`). `src` 직속에 `domain`·`agent` 같은 일반명을 top-level 패키지로 두면 이름 충돌 위험.
- 인터페이스(슬랙 등)를 앱 루트 이름으로 두면 계층이 뒤집힘 → slack은 `client/`의 하나로.
