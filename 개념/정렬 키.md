# 정렬 키 (Sort Key, SK)

같은 [[DynamoDB 파티션 키]] 안에서 아이템을 **정렬·조직하는 축**. PK+SK 복합 키에서 PK가 "어느 파티션"을, SK가 "그 안 어디"를 정한다.

## SK로 할 수 있는 일 4가지
1. **정렬(sort)**: 같은 PK의 아이템이 SK 기준으로 정렬 저장됨. Query `ScanIndexForward=false` → 내림차순(최신순). "최근 주문 10건" 같은 패턴이 자연스럽게 나옴.
2. **범위 조회**: `between`, `>`, `<`, `begins_with` 로 같은 PK 안 범위를 좁힘.
3. **한 파티션에 여러 종류 아이템 조직화** (단일 테이블 설계의 핵심):
   ```
   PK=USER#1, SK=PROFILE
   PK=USER#1, SK=ADDRESS#home
   PK=USER#1, SK=ORDER#2026-06-22
   ```
   한 유저 데이터를 한 파티션(아이템 컬렉션)에 모으고 `begins_with(SK,'ORDER#')`로 주문만 추출. 1:N 관계 모델링 방법.
4. **계층 데이터 조회** (아래).

## 계층 데이터 — begins_with
계층을 구분자(`#`)로 이어 SK에 저장하면, SK가 **문자열로 정렬**되므로 계층 prefix끼리 자동 인접한다.
```
ASIA#JP#TOKYO
ASIA#KR#BUSAN
ASIA#KR#SEOUL
EU#FR#PARIS
```
`begins_with`의 prefix 길이로 조회 깊이를 조절:

| 쿼리 | 결과 |
| --- | --- |
| `begins_with(SK,'ASIA#KR#')` | 한국 모든 도시 |
| `begins_with(SK,'ASIA#')` | 아시아 전체 |

- 계층 상위로 갈수록 prefix가 짧아지고 범위가 넓어진다 → 서브트리를 한 번의 Query로.
- ⚠️ **제약**: prefix는 **왼쪽부터 연속**이어야 한다. "국가 무관 도시명 조회"는 begins_with로 불가 → 도시명을 키로 둔 별도 GSI 필요.

## KeyConditionExpression 구조
- **PK는 반드시 등호(`=`)** — 필수. 파티션을 가로지르는 조회 불가.
- **SK는 선택** — 주면 `=`, `<`, `<=`, `>`, `>=`, `between`, `begins_with` 중 하나.
- 예: `PK = :uid AND begins_with(SK, :prefix)`
- 비-키 속성 필터는 `FilterExpression`이지만, 이건 **데이터를 읽은 뒤 버리는** 방식이라 RCU를 절약하지 못한다 (KeyConditionExpression과 결정적 차이).

#flashcards
- Q: SK의 본질적 역할은? → A: 같은 PK 안에서 아이템을 정렬·조직하는 축.
- Q: 최신순 조회는 어떻게? → A: SK를 시간으로 두고 Query에서 ScanIndexForward=false (내림차순).
- Q: 계층 데이터를 한 번의 Query로 긁는 방법은? → A: 계층을 구분자로 이어 SK에 저장 후 begins_with prefix 길이로 깊이 조절. prefix는 왼쪽부터 연속이어야 함.
- Q: KeyConditionExpression에서 PK와 SK의 연산자 제약은? → A: PK는 반드시 등호, SK는 선택이며 =, 부등호, between, begins_with 가능.
- Q: FilterExpression이 RCU를 절약하지 못하는 이유는? → A: 데이터를 읽은 뒤 필터링해 버리기 때문. 키 조건과 달리 읽기량 자체를 줄이지 못함.
