# 재시도 전략 (Retry)

## 오류 분류 — 무엇을 재시도할 것인가
- **일시적(transient)**: 502, 타임아웃, 일시적 네트워크 → 재시도 **가치 있음**.
- **영구적(permanent)**: 400(잘못된 요청), 파싱 불가 → 재시도해도 같은 실패 = **낭비**.
→ "무조건 N회 재시도"는 안티패턴. 상태코드/오류 종류로 분기해야 한다.

## ⚠️ 재시도 안전성 = 요청의 멱등성
재시도하려면 **그 요청이 멱등**이어야 안전.
- 조회성 GET/POST: 여러 번 보내도 서버 상태 불변 → 안전.
- 비멱등(결제 생성 등): 재시도 = 중복 결제 → 멱등키/중복 검사 선행 필수.
- **재시도를 넣기 전 반드시 물을 질문: "이 요청은 멱등인가?"** ([[멱등성]])

## 백오프 (backoff)
- **횟수**와 **간격**은 다른 축. 횟수만 두고 간격 0이면 = 백오프 없음(즉시 재시도).
- **지수 백오프 + jitter**: 재시도 간격을 점점 늘리고(2^n) 무작위화.
  - 이유: 죽어가는 서버에 재시도 폭주(thundering herd)를 막고 회복 시간을 줌. jitter는 여러 클라이언트의 재시도가 동시에 몰리는 것을 분산.
- 생략 가능한 경우: 모의/즉시응답 환경, 요구가 횟수만 규정. **실서비스는 사실상 필수.**

#flashcards
- Q: transient/permanent 예와 처리? → A: 502=재시도, 400=즉시 실패.
- Q: 재시도 전 반드시 확인할 것? → A: 요청이 멱등인가(비멱등이면 중복 위험).
- Q: 지수 백오프 + jitter를 쓰는 이유? → A: thundering herd 방지 + 서버 회복 시간 + 재시도 시점 분산.
