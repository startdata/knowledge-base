# knowledge-base — 학습·지식 정리 저장소

## 목적
키워드 기반 개념 정리, 조각난 지식을 `[[링크]]`로 연결, 학습·복습, 프로젝트 기록 정리.

## 언어
모든 정리는 한국어로 작성한다.

## 폴더 구조
- `정리/` — 내가 직접 정리·학습하는 내용. 주제별 하위 폴더(예: `cs/`).
- `세션/<주제>/` — Claude 세션에서 다룬 프로젝트·작업 맥락. 프로젝트 기록.
- `개념/` — 재사용 개념의 단일 노트. 여러 세션이 `[[링크]]`로 참조.
- `private/` — 로컬 전용. git에 올리지 않는다(`.gitignore`).

## 개념 노트 (`개념/<키워드>.md`)
- 그 개념의 보편 지식만 — 원리·결론. 프로젝트 디테일·시행착오는 넣지 않는다.
- 같은 개념은 새로 만들지 않고 하나에 누적한다. 이해가 깊어지면 추가한다.
- 완성도: 자동 생성·미완성 노트는 `#stub` 태그를 달고, 모르는 부분은 `?`로 표시한다. 직접 학습으로 채우면 `#stub`을 뗀다. (상태 태그는 `#stub` 하나만 쓴다)

## 세션 노트 (`세션/<주제>/...`) — 프로젝트 기록
프로젝트·기능 작업은 나중에 정리에 쓰므로 과정을 보존한다:
- 문제·배경 — 그때 당시 상태와 제약
- 접근·시행착오 — 무엇을 왜 시도했고, 무엇을 왜 택했나 (로그가 아니라 의사결정 서사로)
- 결과 — 수치화 가능한 포인트 (before/after, 성능·처리량 등)
- 검증 — 어떻게 확인했나
- 사용한 개념 — `[[키워드]]` 링크

## 정리 플로우 (개념의 출처와 무관하게 동일)
개념은 여러 경로로 들어온다 — 세션 대화 중 / 세션 종료·커밋 시 선별 / 직접 학습 / 외부 자료. 출처가 무엇이든 처리는 하나의 파이프라인을 따른다:
1. 해당 `개념/<키워드>.md`에 누적, 없으면 생성 (같은 개념을 새 파일로 만들지 않는다)
2. 보편 지식 → `개념/`, 프로젝트 맥락·시행착오·수치·검증 → `세션/`, 둘을 `[[링크]]`로 연결
3. 주체: 직접 학습이면 내가 먼저 쓰고 Claude가 검증·심화(인출). 자동 정리면 Claude가 선별
4. 핵심 질문은 `#flashcards`로 남긴다 (복습용, 선택)

버림: 무관한 잡담. 깊이 다룬 개념은 노트로, 스쳐간 건 키워드만 남긴다.

## 레퍼런스·출처 규칙 (개념 정리 시)
개념은 신뢰할 출처에 근거해 정리한다. 추측으로 채우지 않는다.
- 검색 우선순위: ① 공식 문서 → ② 공식 문서에 없으면 라이브러리·서비스의 깃 레포 소스코드 확인 → ③ 그래도 없으면 블로그·아티클 참고
- 노트에 출처 링크를 남긴다.
- 공식 문서에 없고 블로그에만 있으면 그 사실을 명시한다: "공식 문서엔 없음 · 블로그([링크]) 출처 — 검증 필요".
- 어디에도 확실한 근거가 없으면: "확실한 레퍼런스 없음. 현재 이해: … (추가 검증 필요)"로 적고 `#stub`을 유지한다.

## 다기기 동기화
집/회사 Mac에서 같은 레포를 쓴다. 작업 시작 시 `git pull`, 종료 시 `git add·commit·push`. (훅으로 자동화)
